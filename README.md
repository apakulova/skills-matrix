# Матрица компетенций редакторов

Интерактивная HTML-страница для оценки навыков редакторов по грейдам. Проект сделан как статический сайт: основная разметка лежит в `index.html`, стили в `styles.css`, данные матрицы и вся логика интерфейса в `script.js`.

Актуальная локальная ссылка для просмотра: `http://127.0.0.1:8000/`.

Публичная ссылка GitHub Pages: `https://apakulova.github.io/skills-matrix/`.

## Что делает сайт

Сайт показывает матрицу компетенций редакторов в виде таблицы:

- слева закрепленный сайдбар с логотипом, фильтрами и счетчиками;
- справа таблица компетенций;
- сверху таблицы закрепленная шапка с колонкой «Компетенции» и грейдами;
- группы навыков оформлены как аккордеоны;
- внутри каждой группы есть строки навыков и ячейки с описанием уровня навыка для каждого грейда.

Пользователь может отмечать каждую ячейку матрицы одним из статусов. Эти отметки используются для подсчета навыков в сайдбаре.

## Грейды

В матрице сейчас используются четыре грейда:

- Junior: «Младший редактор»;
- Middle: «Просто редактор»;
- Senior: «Старший редактор»;
- Lead: «Ведущий редактор».

В фильтре слева есть пять вариантов:

- «Все»;
- Junior;
- Middle;
- Senior;
- Lead.

Если выбран конкретный грейд, остальные колонки не скрываются, а остаются на месте и становятся неактивными. Это сделано специально, чтобы сетка не прыгала при переключении фильтра.

На адаптивных ширинах выбранный грейд переставляется ближе к колонке с названием компетенции. Так пользователю не нужно постоянно скроллить между названием навыка и нужной ячейкой.

## Статусы навыков

У каждой ячейки может быть один из четырех статусов:

- без оценки;
- есть;
- нет;
- так-сяк.

Статус переключается кликом по ячейке по кругу:

`Без оценки -> Есть -> Нет -> Так-сяк -> Без оценки`

Визуальная логика статусов:

- без оценки: белая ячейка;
- есть: зеленая ячейка;
- нет: красная ячейка;
- так-сяк: желтая ячейка.

Названия статусов внутри ячейки не показываются. Это важная договоренность: появление текста статуса раньше меняло высоту строки и ломало спокойную сетку.

Цветной рамки у выбранного навыка тоже нет. Статус выражается только цветом фона.

Оценки хранятся в `localStorage` браузера в ключе `editorMatrixStatuses`. Поэтому отметки сохраняются после перезагрузки страницы на том же устройстве и в том же браузере.

## Как работают фильтры

Когда выбран фильтр «Все»:

- можно отмечать навыки в любых грейдах;
- сайдбар сначала показывает общий счетчик по всей матрице;
- если пользователь начал отмечать навыки, сайдбар показывает блоки по тем грейдам, где уже есть отметки.

Когда выбран конкретный грейд:

- его колонка переставляется ближе к названиям компетенций;
- остальные грейды остаются видимыми, но приглушаются;
- отмечать можно только активный грейд.

Если пользователь случайно отметил навыки в каком-то грейде, рядом с заголовком блока «Навыки Junior», «Навыки Middle» и так далее есть маленький серый крестик. Он сбрасывает все оценки только для этого грейда.

После выбора статуса навыка страница не должна прокручиваться наверх. Пользователь остается в том же месте матрицы, где был до клика. Это важно для длинных групп навыков: иначе оценивать матрицу по одной ячейке было бы неудобно.

## Группы навыков

Группы навыков сделаны аккордеонами. По умолчанию они раскрыты.

Сейчас в проекте есть группы:

- «Работа с текстом»;
- «Работа с информацией»;
- «Продуктовое мышление».

Если группа свернута, строки навыков внутри нее скрываются. В свернутом состоянии у таблицы не должны появляться лишние вертикальные линии между грейдами в пустом пространстве.

## Данные матрицы

Все данные лежат в массиве `rows` в `script.js`.

Внутри массива есть два типа элементов:

- секция-группа навыков;
- строка навыка.

Секция выглядит так:

```js
{ type: "section", id: "product-thinking", title: "Продуктовое мышление" }
```

Строка навыка выглядит так:

```js
{
  id: "ux-ui-understanding",
  title: "Понимание UX и UI",
  kind: "required",
  cells: {
    junior: "...",
    middle: "...",
    senior: "...",
    lead: "...",
  },
}
```

`kind` задает тип навыка:

- `required` - обязательный навык;
- `extra` - дополнительный навык.

Если тип навыка отличается по грейдам, используется объект `kinds`:

```js
kinds: {
  senior: "extra",
  lead: "extra",
}
```

Так сделан, например, «Копирайтинг»: для Junior и Middle он не считается дополнительным навыком, а для Senior и Lead считается.

## Объединенные ячейки из Figma

В Figma некоторые ячейки визуально объединены на несколько грейдов. В коде договоренность другая: у каждого грейда всегда должна быть своя собственная ячейка.

Если в макете текст один и тот же для нескольких грейдов, в `script.js` этот текст дублируется в каждую соответствующую ячейку.

Это нужно, чтобы сайт всегда понимал, к какому грейду относится оценка пользователя. Например, если одинаковый текст подходит Senior и Lead, клик по Senior должен засчитываться Senior, а клик по Lead - Lead.

## Дополнительные навыки

Дополнительный навык может быть:

- всей строкой;
- только ячейкой для конкретного грейда;
- отдельной подстрокой внутри той же компетенции.

Примеры:

- «Копирайтинг» считается дополнительным только для Senior и Lead;
- «Вклад в выполнение целей» имеет отдельную дополнительную подстроку про финансово измеримый результат для Senior и Lead;
- «Сторителлинг» сейчас не считается дополнительным навыком.

Если дополнительный навык относится только к конкретным грейдам, он не должен попадать в счетчик дополнительных навыков для остальных грейдов.

Для Junior и Middle в сайдбаре не показывается строка счетчика дополнительных навыков. В этих грейдах дополнительных навыков по методике нет, поэтому строка вида `0 из 0 дополнительно` считается лишней и только шумит в интерфейсе.

Для Senior и Lead счетчик дополнительных навыков показывается. Сейчас в него попадают:

- «Копирайтинг» как дополнительный навык для Senior и Lead;
- дополнительная подстрока «Вклад в выполнение целей» про финансово измеримый результат для Senior и Lead.

### Подстроки внутри компетенции

Некоторые компетенции могут состоять из основной строки и дополнительной подстроки.

Пример: «Вклад в выполнение целей».

Основная строка есть для всех грейдов и считается как один обычный навык в счетчиках «всего», «обязательно» и «базовый минимум».

Дополнительная подстрока есть только для Senior и Lead. Она не увеличивает общий счетчик навыков и не влияет на базовый минимум, потому что это не отдельная обязательная единица матрицы. При этом она считается в счетчике дополнительных навыков Senior и Lead.

В коде такая подстрока помечается так:

```js
{
  subrow: true,
  countAsSkill: false,
  kinds: {
    senior: "extra",
    lead: "extra",
  },
}
```

## Счетчики в сайдбаре

Счетчики находятся в блоке «Навыки» в левом сайдбаре.

Есть четыре счетчика:

- всего;
- обязательно;
- дополнительно;
- базовый минимум.

У счетчиков «обязательно», «дополнительно» и «базовый минимум» есть маленький вопросик. По наведению, фокусу или клику открывается подсказка с объяснением методики.

Подсказка должна открываться сразу под строкой счетчика, а не внизу экрана и не поверх вопросика. Она имеет фиксированную ширину внутри сайдбара, белый фон, скругление и тень. Подсказка не должна вылезать за границы сайдбара и не должна прятаться под таблицу.

Тексты подсказок:

- обязательные навыки: «Этими навыками должны владеть все редакторы Авито без исключения. Если есть проблемы с такими навыками, обсуди это с руководителем — надо срочно это исправлять.»
- дополнительные навыки: «Каждый старший и ведущий редактор должен владеть хотя бы одним из этих навыков.»
- базовый минимум: «Если хотя бы столько ячеек зелёные — всё отлично.»

### Когда ничего не выбрано и нет отметок

Сайдбар показывает общий счетчик по матрице:

- сколько всего навыков сейчас есть в данных;
- сколько из них обязательные;
- сколько дополнительные;
- базовый минимум.

Числа не должны быть захардкожены вручную. Они считаются из массива `rows`.

### Когда выбран грейд или есть отметки

Счетчик становится счетчиком прогресса по конкретному грейду:

- первая цифра - сколько зеленых навыков у этого грейда;
- вторая цифра - сколько всего навыков есть в соответствующей категории.

Пример:

```text
2 из 17 всего
1 из 12 обязательно
0 из 2 дополнительно
```

В счетчик идут только зеленые ячейки, то есть статус «Есть».

Статусы «Нет» и «Так-сяк» сохраняются и видны цветом, но не увеличивают счетчики.

Если у грейда нет дополнительных навыков, строка «дополнительно» в его блоке не показывается. Сейчас это относится к Junior и Middle.

### Несколько грейдов одновременно

Если фильтр «Все» и пользователь начал отмечать навыки разных грейдов, в сайдбаре появляются отдельные блоки:

- «Навыки Junior»;
- «Навыки Middle»;
- «Навыки Senior»;
- «Навыки Lead».

Первый отмеченный грейд занимает основной блок. Второй и следующие грейды добавляются ниже отдельными секциями.

Если в фильтре выбран конкретный грейд, но в других грейдах уже есть отметки, сайдбар показывает блок выбранного грейда первым, а ниже добавляет блоки остальных отмеченных грейдов. Так пользователь не теряет уже собранную статистику при переключении фильтра.

## Базовый минимум

В матрице есть отдельная договоренность: для хорошего результата нужно 28 зеленых навыков.

Это число хранится в `script.js` как:

```js
const basicMinimum = 28;
```

У счетчика базового минимума три состояния:

1. Если нет выбранного грейда и нет отметок:

   ```text
   28 базовый минимум
   ```

   Используется иконка пустой корзинки.

2. Если у конкретного грейда меньше 28 зеленых навыков:

   ```text
   12 из 28 на базовый минимум
   ```

   Число `12` здесь пример. На сайте оно считается автоматически.

3. Если у конкретного грейда 28 или больше зеленых навыков:

   ```text
   28+ значит базовый минимум есть
   ```

   Используется иконка полного лукошка.

Базовый минимум всегда показывается последним счетчиком в списке.

## Типографика

В проекте согласована система текстовых стилей.

Шапка и логотипный блок используют специальные шрифты:

- русские крупные названия: Panama Monospace;
- английские подписи и маленькая строка «редакторов»: Aeroport Mono.

Остальной интерфейс использует Manrope:

- заголовки разделов;
- названия компетенций;
- фильтры;
- счетчики;
- кнопки;
- описания навыков;
- подписи «Обязательный навык» и «Дополнительный навык».

Текстовые стили заданы CSS-переменными в `styles.css`, чтобы их было проще привести к единой системе.

Тексты также проходят через простой типограф в `script.js`. Он автоматически:

- заменяет прямые кавычки на «елочки»;
- ставит правильные тире в нужных местах;
- добавляет неразрывные пробелы после коротких предлогов и частиц;
- добавляет неразрывные пробелы между числами и короткими единицами.

## Адаптивность

Основные правила адаптивности:

- сайдбар остается слева;
- таблица может горизонтально скроллиться;
- выбранный грейд переставляется ближе к колонке компетенций;
- первая колонка с названиями компетенций может уменьшаться, но слова не должны переноситься внутри слова;
- допустим перенос между словами: «Синтаксическая / стилистика»;
- нежелателен перенос внутри слова: «Синтаксиче- / ская».

Ширина первой колонки и колонок грейдов рассчитывается в `updateMatrixLayoutVariables()`.

## Дизайн-договоренности

Важные решения, которые уже обсуждались:

- не скрывать неактивные грейды, а приглушать их;
- не делать сильный блюр у неактивных колонок;
- не убирать вертикальные границы у приглушенных колонок;
- вертикальное выравнивание ячеек - по верхнему краю;
- у выбранного статуса нет текстовой плашки и цветной рамки;
- нижней панели со статистикой нет, ее заменяет сайдбар;
- горизонтальные разделители между строками нужны, но нижняя граница всей матрицы не должна создавать лишнюю «дырку»;
- между колонками должны быть одинаковые тонкие линии;
- правая граница у колонки Lead в контентной части убрана;
- в шапке вертикальные линии между грейдами остаются;
- стрелка аккордеона стоит рядом с текстом заголовка группы и выровнена по центру текста.

## Контент и Figma

Контент нужно переносить из Figma точно, без самостоятельного переписывания текста.

Если добавляется новая группа:

1. Открыть нужный узел в Figma.
2. Скопировать названия навыков и тексты ячеек в точности из макета.
3. Если в макете ячейка объединена для нескольких грейдов, в коде сделать отдельную ячейку для каждого грейда с тем же текстом.
4. Проверить, какие навыки обязательные, дополнительные или обычные.
5. Добавить группу в `rows`.
6. Добавить `id` группы в `openGroups`, если группа должна быть раскрыта по умолчанию.
7. Проверить счетчики.

Важно: для этого проекта не нужно использовать Code Connect и не нужно пытаться сопоставлять Figma-компоненты с компонентами в коде.

## Локальный запуск

Проект статический, поэтому для локального просмотра достаточно поднять простой сервер из корня проекта:

```bash
python3 -m http.server 8000
```

После этого сайт открывается здесь:

```text
http://127.0.0.1:8000/
```

## Проверки перед показом

Минимальные проверки после изменений:

```bash
node --check script.js
git diff --check
```

После визуальных изменений желательно открыть локальную страницу и проверить:

- шапку;
- сайдбар;
- счетчики;
- переключение фильтра грейдов;
- клики по ячейкам;
- раскрытие и сворачивание групп;
- поведение на более узкой ширине экрана.

apakulova/skills-matrix

## Правила коммитов

В проекте есть `AGENTS.md`. Главное правило: все коммиты писать на русском языке, подробно и простыми словами.

Сообщение коммита должно объяснять:

- что изменилось;
- зачем это было сделано;
- какую часть сайта это затрагивает.

История должна быть понятна без дополнительного контекста даже спустя несколько месяцев.

## Будущее развитие: PDF, ссылки и черновики

Это пока не реализовано. Раздел описывает будущие функции отдельно от текущего состояния сайта: сначала планируем сделать скачивание PDF, потом ссылку-снимок матрицы.

### Что хочется добавить

Внизу левого сайдбара нужны две кнопки:

- «Скачать матрицу как PDF»;
- «Скопировать ссылку».

Обе кнопки не должны попадать в PDF-экспорт. Для таких элементов можно использовать класс `no-export`.

Сайт должен остаться статическим: без бэкенда, базы данных и серверного хранения.

### Скачивание PDF-файла

По PDF уже приняты такие решения:

- не использовать `window.print()`;
- экспортировать всю страницу, кроме элементов с `no-export`;
- сайдбар должен попадать в PDF вместе с матрицей;
- файл должен скачиваться сразу по кнопке;
- PDF должен отражать текущее визуальное состояние сайта;
- если выбран фильтр грейда, выбранный грейд остается первым, остальные грейды остаются приглушенными;
- экспорт всегда должен включать всю ширину матрицы и весь сайдбар, а не только видимую область экрана;
- аккордеоны перед экспортом нужно раскрывать все, чтобы в PDF попала вся матрица целиком;
- подсказки счетчиков в PDF не нужны;
- лишние интерактивные элементы нужно скрывать через `no-export` или временный режим экспорта;
- цвета статусов ячеек должны сохраняться;
- качество должно быть достаточным, чтобы текст хорошо читался;
- сайт должен продолжать работать без бэкенда;
- имя файла должно быть в формате `skills-matrix-ДД-ММ-ГГГГ.pdf`.

Технически решили начать с быстрого варианта: попробовать экспорт DOM через клиентские библиотеки вроде `html2canvas`/`html-to-image` и сборку PDF через `jsPDF`.

Если прототип покажет плохое качество текста, слишком тяжелый файл или проблемы с лимитом высоты canvas на длинной матрице, нужно будет перейти к более надежному варианту: генерировать PDF из данных матрицы через `pdf-lib` или `jsPDF`, вручную рисуя сетку, тексты, цвета и изображения.

План PDF-реализации:

1. Добавить в сайдбар кнопку «Скачать матрицу как PDF».
2. Дать кнопкам внизу сайдбара класс `no-export`.
3. Добавить временный режим экспорта, например класс `is-exporting` на `body`.
4. В режиме экспорта скрывать `.no-export`, открытые подсказки и лишние интерактивные состояния.
5. Перед экспортом временно раскрывать все аккордеоны.
6. Убедиться, что экспортируется вся ширина матрицы и весь сайдбар.
7. Сформировать имя файла по текущей дате в формате `skills-matrix-ДД-ММ-ГГГГ.pdf`.
8. Сделать быстрый прототип DOM-экспорта.
9. Проверить читаемость текста, вес файла, длинную высоту PDF и корректность цветов.
10. Если качество не подойдет, заменить технический подход на ручную генерацию PDF из данных.

Перед стартом PDF-работы открытых продуктовых вопросов сейчас нет. Остался только технический риск качества быстрого DOM-экспорта, который нужно проверить прототипом.

### Ссылка-снимок матрицы: что уже понятно

Кнопка «Скопировать ссылку» должна создавать ссылку, в которой зашито текущее состояние матрицы:

```text
https://site.ru/#matrix=...
```

В ссылке не нужно хранить тексты компетенций. Нужно хранить только состояние:

- версию формата;
- выбранный грейд;
- статусы отмеченных ячеек;
- стабильные id навыков;
- id грейдов;
- возможно, имя сотрудника;
- возможно, состояние раскрытых и свернутых групп.

Ссылка должна быть снимком, а не живым документом. Если пользователь открыл ссылку, изменил матрицу и хочет сохранить новую версию, он должен снова нажать «Скопировать ссылку».

В ссылке не нужно хранить ячейки без оценки. Достаточно хранить только статусы:

- `Есть`;
- `Нет`;
- `Так-сяк`.

Статус «Без оценки» можно считать отсутствием записи.

Для компактности можно кодировать состояние короткими значениями, например `1`, `2`, `3`, а затем сжимать объект в URL-safe строку. Один из возможных вариантов - использовать клиентскую библиотеку `lz-string`, положенную локально в проект.

По ссылке пока решение отложено: сначала делаем PDF, затем возвращаемся к ссылке-снимку.

### Стабильные ID

Для ссылок особенно важно, чтобы у каждого навыка был стабильный id, не зависящий от порядка строк в таблице.

Правильный id должен быть смысловым, например:

```text
grammar
copywriting
lexical-style
```

Нежелательные id:

```text
row-1
skill-2
item-17
```

Если порядок навыков поменяется, старая ссылка должна продолжить работать. Если когда-нибудь понадобится переименовать id, стоит добавить слой алиасов для старых ссылок.

### Автосохранение и черновики

Автосохранение нужно, чтобы пользователь не потерял отметки после обновления страницы или случайного закрытия вкладки.

Сейчас отметки уже сохраняются в `localStorage`, но для будущей логики ссылок лучше оформить это как отдельную модель состояния:

```js
getMatrixState()
applyMatrixState(state)
saveMatrixState()
loadMatrixState()
```

Есть проблема: если человек открыл ссылку коллеги, сайт не должен незаметно стереть его собственный локальный черновик.

Поле имени сотрудника может помочь, но само по себе не решает проблему полностью. В браузере все равно один `localStorage` для сайта. Чтобы разные матрицы не конфликтовали, лучше хранить черновики по техническому `draftId`.

Рабочая идея:

```js
{
  v: 1,
  draftId: "m8x4k2",
  name: "Анна",
  grade: "junior",
  statuses: {}
}
```

Имя нужно для человека, а `draftId` - для надежного технического различения черновиков.

Минимальная модель без сложного интерфейса:

- хранить несколько черновиков в `localStorage` по `draftId`;
- хранить последний активный `draftId`;
- не делать пока список черновиков в интерфейсе;
- если открыли ссылку, применить ее состояние на экран;
- не удалять автоматически старый локальный черновик;
- после изменения открытого снимка сохранять его как черновик по его `draftId`.

### Примерный порядок реализации ссылок и черновиков

1. Проверить и стабилизировать все `row.id` в данных матрицы.
2. Ввести единый формат состояния матрицы.
3. Перевести `localStorage` на новую модель состояния.
4. Добавить поле имени сотрудника, если решим, что оно нужно в первой итерации.
5. Добавить `draftId` и модель нескольких локальных черновиков без отдельного UI для списка.
6. Добавить кодирование и декодирование состояния в `#matrix=...`.
7. При загрузке страницы читать состояние из URL, если оно есть.
8. Добавить кнопку «Скопировать ссылку» и обратную связь после копирования.
9. Обновить README после реализации, чтобы документация описывала уже не план, а фактическую работу.

### Открытые вопросы по ссылке

Перед реализацией ссылки нужно решить:

- добавляем ли поле имени сотрудника сразу;
- как назвать это поле в интерфейсе;
- где разместить поле имени;
- должно ли имя попадать в PDF;
- должно ли имя попадать в ссылку;
- нужна ли модель нескольких черновиков по `draftId` уже в первой итерации;
- нужно ли показывать уведомление, что открыта матрица из ссылки, а локальный черновик не удален;
- сохранять ли в ссылке состояние раскрытых и свернутых групп;
- можно ли добавлять клиентские библиотеки в проект и хранить их локально.
